Symfony 3.3 shipped May 29th. It's the release that changed how I think about service configuration — and in hindsight, it was the preview of the architecture that 4.0 would make default.

## :hammer_and_wrench: The autowiring problem

Before 3.3, Symfony's dependency injection was powerful but verbose. Every service had to be declared explicitly in `services.yml` with its arguments listed. Autowiring existed in 3.1, but it was opt-in per service and had edge cases that bit people. Teams either wrote mountains of YAML or relied on third-party bundles to reduce the noise.

3.3 rewrote the defaults. With `autoconfigure: true` and `autowire: true` set once in the defaults section, every class in `src/` becomes a service automatically, and its constructor dependencies are resolved by type. What used to take twenty lines of YAML now takes zero:

```yaml
services:
    _defaults:
        autowire: true
        autoconfigure: true
    App\:
        resource: '../src/'
```

That single block is the entire service configuration for most applications. The framework discovers services, injects dependencies, and applies tags (command, event subscriber, voter...) based on the interfaces each class implements.

## :card_index_dividers: instanceof conditionals

The `instanceof` keyword in service configuration handles the tagging that previously required explicit declaration:

```yaml
services:
    _instanceof:
        Symfony\Component\EventDispatcher\EventSubscriberInterface:
            tags: ['kernel.event_subscriber']
```

Any service implementing `EventSubscriberInterface` gets the tag automatically. Same for `Command`, `Voter`, `MessageHandlerInterface`. The boilerplate evaporates.

## :page_facing_up: Dotenv component

Symfony had no native way to load `.env` files before 3.3. The standard was a third-party package. The new `Dotenv` component reads `.env` and populates `$_ENV` and `$_SERVER`, making environment-based configuration a first-class citizen.

## :crystal_ball: What this meant for 4.0

3.3's autowiring defaults are exactly what Symfony 4.0 shipped as the new standard project structure. The services.yaml in every new Symfony 4 project is essentially the snippet above. If you understood what 3.3 introduced, 4.0's "new way" was already familiar.

The direction was clear: less configuration, more convention. PHP would figure out what to wire together.
