Symfony 3.4 and 4.0 were released the same day — November 30, 2017. That's not a coincidence; it's the strategy.

3.4 is not a feature release. It ships with exactly the same features as 3.3 plus the new deprecation warnings from 4.0. Its purpose is to be the migration tool: upgrade from 3.3 to 3.4, fix every deprecation warning in your logs, then step to 4.0 cleanly.

## :bridge_at_night: Why LTS releases matter in Symfony's model

Symfony releases a new minor version every six months. That pace would be exhausting for production applications to follow, so the project designates every fourth minor as an LTS: three years of bug fixes, four of security fixes. The practical effect is that teams can target 3.4 and not think about upgrades for years.

3.4 is the last LTS of the 3.x line. If you're still on Symfony 2.x or early 3.x, 3.4 is the landing zone.

## :warning: The deprecation layer

Every feature that 4.0 removes is deprecated in 3.4. Run your application on 3.4, enable the deprecation notices, and your logs tell you exactly what to fix. The common ones:

- Services without explicit visibility (public/private) generate warnings — 4.0 makes all services private by default
- `ControllerTrait` is deprecated in favor of `AbstractController`
- The old security authenticator interfaces are marked for removal
- YAML-only service configuration without autowiring annotations triggers warnings

The workflow Symfony intended: upgrade to 3.4, run the test suite with deprecation notices as errors (`SYMFONY_DEPRECATIONS_HELPER=max[self]=0` in PHPUnit), fix everything that fails, then upgrading to 4.0 is mechanical.

## :calendar: The support window

3.4 LTS receives bug fixes until November 2020 and security fixes until November 2021. That's a long runway for production applications that can't follow every release. The cost is staying on the 3.x architecture — no Flex, no micro-framework structure, no zero-config autowiring by default.

The bridge is there. Whether and when you cross it is a business decision, not a technical one.

## Services go private

3.4 flipped the default visibility of services from public to private. Before this, `$container->get('app.my_service')` was normal code. After this, it is an anti-pattern that generates a deprecation warning in 3.4 and breaks entirely in 4.0.

The reasoning is straightforward: fetching services directly from the container hides dependencies and defeats static analysis. If you inject dependencies through the constructor, the container can optimize the graph, tree-shake unused services, and catch mistakes at compile time. If you pull them at runtime, it cannot.

For applications already using autowiring, the migration is usually small. The sticking point is controllers that extend `Controller` and call `$this->get('something')`. The path forward is switching to `AbstractController`, which provides a curated set of shortcuts while injecting dependencies properly.

For services that genuinely need to be public (accessed from legacy code or functional tests), mark them explicitly:

```yaml
services:
    App\Service\LegacyAdapter:
        public: true
```

## Binding scalar arguments once

A recurring friction point with autowiring was scalar constructor arguments. If ten services all need `$projectDir`, you had to configure each one individually. The `bind` key under `_defaults` solves this:

```yaml
services:
    _defaults:
        autowire: true
        autoconfigure: true
        bind:
            $projectDir: '%kernel.project_dir%'
            $mailerDsn: '%env(MAILER_DSN)%'
            Psr\Log\LoggerInterface $auditLogger: '@monolog.logger.audit'
```

Any service with a constructor parameter named `$projectDir` receives the bound value automatically. You can also bind by type-hint, which handles the common case where multiple logger channels exist and you need to inject a specific one. Bindings in `_defaults` apply to all services in the file; you can override them per-service if needed.

## Injecting tagged services without a compiler pass

Before 3.4, collecting all services with a given tag into a single service required writing a compiler pass. Now there is a YAML shorthand:

```yaml
services:
    App\Chain\TransformerChain:
        arguments:
            $transformers: !tagged app.transformer
```

```php
class TransformerChain
{
    public function __construct(private iterable $transformers) {}
}
```

The `!tagged` notation creates an `IteratorArgument` — the services are lazily instantiated as you iterate, so unused transformers in a chain do not get built. For ordering, add a `priority` attribute to the tag definition on each service.

## A logger that ships with the framework

If you do not install Monolog, Symfony 3.4 provides a PSR-3 logger that writes to `php://stderr` by default. Autowire it with `Psr\Log\LoggerInterface`:

```php
use Psr\Log\LoggerInterface;

class MyService
{
    public function __construct(private LoggerInterface $logger) {}

    public function doSomething(): void
    {
        $this->logger->warning('Something questionable happened', ['context' => 'here']);
    }
}
```

The default minimum level is `warning`. The intent is containers and Kubernetes workloads where stderr is the natural log sink. It is deliberately minimal — no handlers, no processors, no channels. Install Monolog when you need those.

## Guard authenticators got a supports() method

The Guard component's `getCredentials()` method was doing two things at once: deciding whether the authenticator should handle the request, and extracting the credentials. Returning `null` from `getCredentials()` was the signal to skip the authenticator. That made the method's contract unclear.

3.4 introduced `supports()` to separate those concerns:

```php
class ApiTokenAuthenticator extends AbstractGuardAuthenticator
{
    public function supports(Request $request): bool
    {
        return $request->headers->has('X-API-TOKEN');
    }

    public function getCredentials(Request $request): array
    {
        // Only called when supports() returns true.
        // Must always return credentials now.
        return ['token' => $request->headers->get('X-API-TOKEN')];
    }
}
```

The old `GuardAuthenticatorInterface` is deprecated. The practical benefit of the split is that base classes can implement shared `getUser()` and `checkCredentials()` logic while subclasses only override `supports()` and `getCredentials()`, one responsibility each.

## Two new debug commands

`debug:autowiring` replaces the awkward `debug:container --types` for discovering what type-hints are usable with autowiring:

```bash
$ bin/console debug:autowiring log

Autowirable Services
====================

  Psr\Log\LoggerInterface
      alias to monolog.logger
  Psr\Log\LoggerInterface $auditLogger
      alias to monolog.logger.audit
```

Pass a keyword to filter. No more guessing whether the interface is `LoggerInterface` or `Logger`.

`debug:form` gives you the same introspection capability for form types:

```bash
$ bin/console debug:form App\Form\OrderType label_attr

Option: label_attr
  Required: false
  Default: []
  Allowed types: array
```

Without arguments it lists all registered form types, extensions, and guessers. With a type name and option name it shows every constraint on that option. Previously you either read the source or trial-and-errored your way through.

## Sessions got stricter by default

3.4 implements PHP 7.0's `SessionUpdateTimestampHandlerInterface`, which enables two things: lazy session writes (the session is only written when data actually changed) and strict session ID validation (session IDs that do not exist in the store are rejected rather than silently created, which prevents a class of session fixation attacks).

The old `WriteCheckSessionHandler`, `NativeSessionHandler`, and `NativeProxy` classes are deprecated. The `MemcacheSessionHandler` (note: not Memcached) is also deprecated because the underlying PECL extension stopped receiving PHP 7 updates.

## Twig form themes can now be scoped

Global form themes apply to every form in the application. If you have a special form that needs a completely different look, you had no clean way to opt out. The `only` keyword handles this:

```twig
{% raw %}{% form_theme orderForm with ['form/order_layout.html.twig'] only %}{% endraw %}
```

The `only` keyword disables all global themes for that form, including the base `form_div_layout.html.twig`. Your custom theme needs to either provide all the blocks it uses or pull them in explicitly with `{% raw %}{% use 'form_div_layout.html.twig' %}{% endraw %}`.

## Overriding bundle templates without infinite loops

Overriding a bundle template that you also need to extend used to produce a circular reference error. If you wanted to override `@TwigBundle/Exception/error404.html.twig` and also inherit from it, the old namespace resolution would follow your override and create a loop.

3.4 introduced the `@!` prefix to explicitly reference the original bundle template, bypassing any overrides:

```twig
{% raw %}{# templates/bundles/TwigBundle/Exception/error404.html.twig #}
{% extends '@!Twig/Exception/error404.html.twig' %}

{% block title %}Page not found{% endblock %}{% endraw %}
```

`@TwigBundle` resolves to your override if one exists. `@!TwigBundle` always resolves to the original bundle. This makes it possible to override-and-extend without gymnastics.
