PHP 8.0 shipped November 26th. I've been using it for six weeks on a side project and a greenfield service at work. It's the most significant PHP release since 7.0 — and in some ways more impactful, because the changes compound each other.

## :zap: JIT

The Just-In-Time compiler was the headline announcement. The reality in production is nuanced: for typical web applications — database queries, HTTP calls, template rendering — the gains are modest because those workloads are I/O bound, not compute bound. Where JIT shines is CPU-intensive code: image manipulation, data transformation, mathematical computation.

For most web apps, the performance improvement comes from the overall engine improvements in 8.0, not JIT specifically. Still worth enabling; it costs nothing on I/O-bound work.

## :twisted_rightwards_arrows: Match expressions

`switch` has three problems: it uses loose comparison, it falls through by default, and it can't be used as an expression. `match` fixes all three:

```php
$result = match($status) {
    'active', 'pending' => 'processing',
    'done'             => 'finished',
    default            => throw new \UnexpectedValueException($status),
};
```

Strict comparison. No fall-through. Expression that returns a value. Non-exhaustive match throws. After one week with `match` I stopped writing `switch`.

## :pushpin: Named arguments

```php
array_slice(array: $users, offset: 0, length: 10, preserve_keys: true);
```

Named arguments let you pass arguments in any order and skip optional ones. The immediate benefit is readability on functions with multiple boolean flags. The less obvious benefit: named arguments are stable across PHP version upgrades even when parameter order changes, because the name is explicit.

## :label: Attributes

Replacing docblock annotations — the `@Route`, `@ORM\Column` style that frameworks have used for years — with first-class PHP syntax:

```php
#[Route('/users', methods: ['GET'])]
#[IsGranted('ROLE_ADMIN')]
public function list(): Response { ... }
```

Attributes are validated by the engine, not parsed from strings. IDE support is immediate and accurate. For Symfony and Doctrine users this is the practical daily win of PHP 8.0.

## :construction_worker: Constructor promotion

```php
class User {
    public function __construct(
        public readonly int $id,
        public string $name,
        private ?string $email = null,
    ) {}
}
```

Properties declared and assigned in one line in the constructor signature. The most immediate refactoring win in 8.0 — every data class I've touched since upgrading is half the lines it used to be.

## :question: Nullsafe operator

```php
$city = $user?->getAddress()?->getCity()?->getName();
```

`null` at any point in the chain short-circuits the rest and returns `null`. The alternative was nested null checks or a chain of early returns. This composes naturally.

## Union types

Named arguments make function signatures more explicit at the call site. Union types make them more honest at the declaration site:

```php
function processInput(int|float|string $value): string|int
{
    if (is_string($value)) {
        return strlen($value);
    }
    return (int) round($value);
}
```

The union `int|float|string` is a literal OR. The engine enforces it on entry and exit. Before 8.0, "this parameter accepts int or float" lived in a docblock that nothing enforced. There's also `null` as a type component, which makes `?string` syntactic sugar for `string|null` — same thing, both are valid.

One special union: `false`. PHP has several built-in functions that return a typed value on success and `false` on failure. The 8.0 type system accommodates that: `array|false`, `string|false`. It's an acknowledgment that the codebase can't be rewritten overnight.

## static return type

`static` as a return type was possible informally through docblocks, but 8.0 makes it first-class. The distinction between `self` and `static` matters in inheritance:

```php
class Builder {
    protected array $config = [];

    public function set(string $key, mixed $value): static {
        $this->config[$key] = $value;
        return $this;
    }
}

class SpecialBuilder extends Builder {}

$result = (new SpecialBuilder())->set('foo', 'bar');
// $result is SpecialBuilder, not Builder
```

With `self` as the return type, that chain would return `Builder` — breaking fluent interfaces in subclasses. `static` makes fluent APIs work correctly across inheritance hierarchies without manual overrides.

## mixed type

`mixed` existed as a docblock convention for years. 8.0 makes it a proper type that can appear in signatures:

```php
function debug(mixed $value): void {
    var_dump($value);
}
```

It accepts everything: `null`, objects, resources, scalars, arrays. It's semantically equivalent to no type declaration at all, but it's explicit rather than absent — the difference between "this parameter is untyped" and "this parameter intentionally accepts anything." Worth using when you're implementing a general-purpose utility that would be dishonest with a narrower type.

## throw as expression

Before 8.0, `throw` was a statement. That sounds like a pedantic distinction until you hit the places where you want an expression:

```php
// In a ternary:
$value = $input ?? throw new \InvalidArgumentException('input required');

// In an arrow function:
$getId = fn(User $u) => $u->id ?? throw new \RuntimeException('no id');

// In a match arm (which is already an expression):
$status = match($code) {
    200     => 'ok',
    404     => 'not found',
    default => throw new \UnexpectedValueException("unknown code: $code"),
};
```

The last one is particularly useful — match without a default will throw `UnhandledMatchError` automatically, but sometimes you want to control the exception type and message.

## catch without a variable

Small quality-of-life fix. When you catch an exception but don't actually use the object, 8.0 lets you omit the variable:

```php
try {
    $result = $cache->get($key);
} catch (CacheMissException) {
    $result = $this->compute($key);
}
```

Before 8.0, you had to write `catch (CacheMissException $e)` and then either use `$e` or live with the IDE warning about an unused variable. Neither was satisfying.

## String functions that should have existed years ago

Three functions that every PHP developer has written manually at least once:

```php
str_contains('hello world', 'world');  // true
str_starts_with('hello world', 'hell'); // true
str_ends_with('hello world', 'world'); // true
```

Before 8.0, the idiomatic ways were `strpos() !== false`, `strncmp()`, or `substr() ===` — all of which require stopping to think about the semantics. These are direct and readable. No regex, no offset arithmetic.

## Stable sort

PHP's sorting functions were not stable before 8.0. "Not stable" means elements that compare as equal could end up in any order relative to each other. In practice this caused subtle bugs in UI code that needed consistent ordering, pagination that shifted between loads, and tests that depended on sort order being deterministic.

8.0 guarantees stability across all sorting functions: `sort()`, `usort()`, `array_multisort()`, and the rest. Equal elements preserve their original relative position. This is the behavior most people assumed was already there.

## WeakMap

7.4 brought `WeakReference` for single objects. 8.0 brings `WeakMap` — a map where both the keys (objects) and their associated data can be garbage collected when no other reference to the key object exists:

```php
class RequestCache {
    private WeakMap $cache;

    public function __construct() {
        $this->cache = new WeakMap();
    }

    public function get(Request $request): Response {
        return $this->cache[$request] ??= $this->compute($request);
    }
}
```

The moment `$request` is no longer referenced anywhere else, the entry disappears from the map. No manual cleanup. This is the pattern behind memoization and computed property caches where you don't want to be the sole reason an object stays alive.

## New exception types

`ValueError` is thrown when a function receives a value of the right type but an invalid value — distinguishing from `TypeError` which fires on wrong types:

```php
array_chunk([], -5); // ValueError: array_chunk(): Argument #2 ($length) must be greater than 0
```

Before 8.0, many of these were warnings that returned `false` or `null`. Now they're exceptions. The engine is stricter, which means errors surface earlier rather than silently producing wrong results downstream.

## get_debug_type() and fdiv()

Two utility functions with specific purposes.

`get_debug_type()` returns a normalized string representation of any value, suitable for error messages:

```php
get_debug_type(1);          // "int"
get_debug_type(1.0);        // "float"
get_debug_type(null);       // "null"
get_debug_type(new Foo());  // "Foo" (not "object")
get_debug_type([]);         // "array"
```

The difference from `gettype()`: it returns class names for objects and normalized names (`"int"` not `"integer"`). Useful when you're writing an exception message that describes what you received versus what you expected.

`fdiv()` performs floating-point division following IEEE 754 — which means division by zero returns `INF`, `-INF`, or `NAN` instead of a warning:

```php
fdiv(10, 0);   // INF
fdiv(-10, 0);  // -INF
fdiv(0, 0);    // NAN
```

## The changes that break things

8.0 ships with a few changes that aren't features — they're corrections.

The most impactful: `0 == "foo"` is now `false`. In PHP 7, comparing an integer to a non-numeric string cast the string to 0. This meant `0 == "anything-non-numeric"` was `true`, which was a persistent source of bugs and security issues. PHP 8 casts the integer to a string for the comparison instead:

```php
var_dump(0 == "foo");  // bool(false) in 8.0, bool(true) in 7.x
var_dump(0 == "");     // bool(false) in 8.0, bool(true) in 7.x
var_dump(0 == "0");    // bool(true) in both — "0" is numeric
```

If you relied on this behavior intentionally, you already knew it was fragile. If you didn't know you relied on it, 8.0 will find those code paths for you.

Several functions that used to return resources now return proper objects: `curl_init()` returns a `CurlHandle`, `imagecreate()` returns a `GdImage`, `xml_parser_create()` returns an `XMLParser`. Code that does `is_resource($curl)` will break — `is_resource()` returns `false` for these objects. The fix is checking against `false` (the return value on failure) rather than checking the type of the success case.

PHP 8.0 is the kind of release where the features reinforce each other. Attributes work better with constructor promotion. Match works better with union types. The string functions eliminate noise that obscured intent. The corrections, while occasionally breaking, push the language toward consistency it should have had earlier.
