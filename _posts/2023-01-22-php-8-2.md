PHP 8.2 dropped December 8th. Readonly classes are the headline feature. The deprecation of dynamic properties is the one that requires attention.

## :no_entry_sign: Dynamic properties deprecated

PHP has always allowed adding properties to objects that weren't declared in the class:

```php
class User {}
$user = new User();
$user->name = 'Alice'; // no declaration, no error — until now
```

In 8.2, this triggers a deprecation notice. In PHP 9.0 it becomes a fatal error. The grace period is there, but the migration clock is running.

The rationale is sound: dynamic properties are a common source of typos that silently pass (if you write `$user->nmae`, PHP creates a new property instead of throwing). Explicit property declarations make the class contract clear and tooling analysis reliable.

Migration is mostly mechanical: declare the properties, or use `#[AllowDynamicProperties]` for legacy classes you can't touch yet.

## :pencil: Readonly classes

8.1 added `readonly` for individual properties. 8.2 adds it to the class declaration itself:

```php
readonly class Point {
    public function __construct(
        public float $x,
        public float $y,
        public float $z,
    ) {}
}
```

All promoted and explicitly declared properties become readonly automatically. Value objects — coordinates, money amounts, identifiers — are the target use case. The syntax is clean and the intent is explicit.

One constraint: readonly classes can't have non-typed properties (which were already problematic with readonly anyway).

## :twisted_rightwards_arrows: DNF types

Disjunctive Normal Form types allow combining union and intersection types:

```php
function process(Countable&Iterator|null $collection): void { ... }
```

`(Countable&Iterator)|null` — an object that implements both interfaces, or null. This unlocks type expressions that 8.0 union types and 8.1 intersection types couldn't represent together.

## :game_die: The Random extension

A dedicated `Random` extension replaces the scattered `rand()`, `mt_rand()`, `random_int()` functions with an object-oriented API:

```php
$rng = new Random\Randomizer();
$rng->getInt(1, 100);
$rng->shuffleArray($items);
```

Engines are swappable: `Mersenne Twister`, `PCG64`, `Xoshiro256StarStar`, or `CryptoSafeEngine` for security-sensitive contexts. The same code can use a seeded deterministic engine in tests and a cryptographic engine in production.

8.2 is a consolidation release. The dynamic properties deprecation is the decision you need to make now.

## :label: `null`, `false`, and `true` as standalone types

PHP has had `nullable` types since 7.1 and union types since 8.0, but `null` as a standalone type declaration wasn't valid. 8.2 fixes that:

```php
function alwaysNull(): null {
    return null;
}

function disabled(): false {
    return false;
}

function enabled(): true {
    return true;
}
```

`false` and `true` as standalone types are useful when you need to be precise about a function that can only return one of those values. It's narrow but correct — a function that returns `false` on failure and a string on success should declare `string|false`, and now both sides of that union are real types.

## :package: Constants in traits

Traits could hold properties and methods. Constants were the gap. 8.2 closes it:

```php
trait Timestamps {
    public const DATE_FORMAT = 'Y-m-d H:i:s';

    public function formatCreatedAt(): string {
        return $this->createdAt->format(self::DATE_FORMAT);
    }
}

class Article {
    use Timestamps;
}

echo Article::DATE_FORMAT; // 'Y-m-d H:i:s'
```

The constant belongs to the class that uses the trait, not to the trait itself — you can't access `Timestamps::DATE_FORMAT` directly. That's the expected scoping behavior, consistent with how trait methods work.

## :zipper_mouth_face: `#[SensitiveParameter]`

Stack traces have always been a liability: function arguments get logged verbatim, which means passwords and tokens end up in error logs and monitoring platforms. 8.2 adds an attribute to prevent that:

```php
function authenticate(
    string $user,
    #[\SensitiveParameter] string $password,
): bool {
    // if this throws, the stack trace shows:
    // authenticate('alice', Object(SensitiveParameterValue))
    return hash('sha256', $password) === getStoredHash($user);
}
```

The parameter value in the trace is replaced with a `SensitiveParameterValue` object. The fix is one attribute, and there's no reason not to apply it to every function that touches credentials.

## :scissors: Deprecated string interpolation syntaxes

Two ways to interpolate expressions inside strings are deprecated in 8.2:

```php
$name = 'world';

// These are deprecated:
echo "Hello ${name}";       // use "$name" or "{$name}"
echo "Hello ${getName()}";  // use "{$this->getName()}"
```

The `${...}` forms created ambiguity between variable variables and expressions. The cleaner `{$...}` syntax has always been there and does the same thing. This is mostly a search-and-replace migration in codebases that adopted the deprecated forms habitually.

## :wastebasket: `utf8_encode()` and `utf8_decode()` deprecated

These two functions are deprecated in 8.2 and gone in 9.0. Their behavior was always narrower than their names implied: `utf8_encode()` converts ISO-8859-1 to UTF-8, not "any encoding to UTF-8."

```php
// Deprecated in 8.2:
$utf8 = utf8_encode($latin1String);

// Use instead:
$utf8 = mb_convert_encoding($latin1String, 'UTF-8', 'ISO-8859-1');
```

`mb_convert_encoding()` or `iconv()` handle the general case. If you're actually dealing with Latin-1 input, the replacement is a direct swap.

## :globe_with_meridians: Locale-independent string functions

Several string functions that silently varied behavior based on the system locale — producing different results in production versus a dev container — are now locale-independent and ASCII-only in 8.2:

```php
// strtolower, strtoupper, stristr, stripos, strripos,
// lcfirst, ucfirst, ucwords, str_ireplace now do ASCII case conversion only.
// For locale-aware behavior, use mb_* equivalents:
$lowered = mb_strtolower($text, 'UTF-8');
```

This is a correctness fix. If your code was relying on locale-sensitive behavior from these functions, it was already broken on systems with different locale configurations. The 8.2 change makes the behavior deterministic everywhere.

## :arrows_counterclockwise: `str_split()` on empty string

A quiet behavior change worth noting:

```php
// PHP 8.1: str_split('') === ['']
// PHP 8.2: str_split('') === []
```

The new behavior is more consistent — splitting nothing produces nothing. If you're checking `count(str_split($input))`, an empty input no longer produces a count of 1.
