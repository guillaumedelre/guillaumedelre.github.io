PHP 8.4 released November 21st. Property hooks are the feature. Everything else — and there's quite a bit of it — is secondary.

## :hook: Property hooks

For twenty years, adding behavior to property access in PHP meant writing getters and setters:

```php
class User {
    private string $_name;
    
    public function getName(): string { return $this->_name; }
    public function setName(string $name): void {
        $this->_name = strtoupper($name);
    }
}
```

PHP 8.4 adds hooks directly on the property:

```php
class User {
    public string $name {
        set(string $name) {
            $this->name = strtoupper($name);
        }
    }
}
```

You can define `get` and `set` hooks independently. A property with only a `get` hook is effectively computed:

```php
class Circle {
    public float $area {
        get => M_PI * $this->radius ** 2;
    }
    
    public function __construct(public float $radius) {}
}
```

No backing storage, no explicit getter method, full IDE support. Interfaces can declare properties with hooks, which means contracts can now specify behavior on property access — something that was impossible before.

## :eyes: Asymmetric visibility

A lighter alternative to hooks when you just want public read, private write:

```php
class Version {
    public private(set) string $value = '1.0.0';
}

$v = new Version();
echo $v->value;      // works
$v->value = '2.0';  // Error
```

Stops the `private $x` + `public getX()` pattern for read-only public properties without needing full readonly semantics.

## :mag: array_find() and friends

```php
$first = array_find($users, fn($u) => $u->isActive());
$any   = array_any($users, fn($u) => $u->isPremium());
$all   = array_all($users, fn($u) => $u->isVerified());
```

These have existed in every other language's standard library for decades. In PHP, the equivalent required `array_filter()` + index access or a manual loop. The functions exist now — `array_find()`, `array_find_key()`, `array_any()`, `array_all()`.

## :new: Instantiation without extra parentheses

```php
// before
(new MyClass())->method();

// after
new MyClass()->method();
```

A syntax restriction that existed for no good reason is gone.

## :zzz: Lazy objects

Objects whose initialization is deferred until first property access:

```php
$user = $reflector->newLazyProxy(fn() => $repository->find($id));
// No database call yet
$user->name; // Now the proxy initializes
```

The primary audience is framework ORM and DI container authors, not application developers directly. But the effect is felt in every application that uses Doctrine or Symfony — lazy loading can be implemented at the language level rather than through code generation.

PHP 8.4 is a language that looks different from the PHP 5 many developers started with. Property hooks particularly — they're not a workaround, they're a design feature.

## #[\Deprecated] for your own code

PHP has had deprecation notices for built-in functions for years. 8.4 lets you attach the same mechanism to your own code:

```php
class ApiClient {
    #[\Deprecated(
        message: 'Use fetchJson() instead',
        since: '2.0',
    )]
    public function get(string $url): string { ... }
}
```

Calling a deprecated method now emits `E_USER_DEPRECATED`, just like calling `mysql_connect()`. IDEs pick it up, static analyzers flag it, and the error log captures it. The only way to mark deprecated code before was a `@deprecated` PHPDoc comment — functional for IDEs but invisible to the engine.

## BcMath\Number makes arbitrary precision usable

The `bcmath` functions have been in PHP since forever, but their procedural API makes chaining operations painful. 8.4 adds `BcMath\Number`, an object wrapper that supports operator overloading:

```php
$a = new BcMath\Number('10.5');
$b = new BcMath\Number('3.2');

$result = $a + $b;             // BcMath\Number('13.7')
$result = $a * $b - new BcMath\Number('1');
echo $result;                  // 32.6
```

The `+`, `-`, `*`, `/`, `**`, `%` operators all work. The object is immutable. Scale propagates automatically through operations. Financial calculations, which previously required chains of `bcadd(bcmul(...), ...)`, now read like arithmetic.

New procedural functions complete the picture: `bcceil()`, `bcfloor()`, `bcround()`, `bcdivmod()`.

## RoundingMode enum replaces PHP_ROUND_* constants

`round()` has always taken a `$mode` int from a set of `PHP_ROUND_*` constants. 8.4 introduces a `RoundingMode` enum with cleaner names and four additional modes not previously available:

```php
round(2.5, mode: RoundingMode::HalfAwayFromZero);  // 3
round(2.5, mode: RoundingMode::HalfTowardsZero);   // 2
round(2.5, mode: RoundingMode::HalfEven);          // 2 (banker's rounding)
round(2.5, mode: RoundingMode::HalfOdd);           // 3

// The four new modes (only available via the enum)
round(2.3, mode: RoundingMode::TowardsZero);       // 2
round(2.7, mode: RoundingMode::AwayFromZero);      // 3
round(2.3, mode: RoundingMode::PositiveInfinity);  // 3
round(2.3, mode: RoundingMode::NegativeInfinity);  // 2
```

The old `PHP_ROUND_*` constants still work. The enum is the path forward.

## Multibyte string functions that should have existed

`mb_trim()`, `mb_ltrim()`, `mb_rtrim()` — trim functions that respect multibyte character boundaries, not just ASCII whitespace. Also new: `mb_ucfirst()` and `mb_lcfirst()` for proper title-casing of multibyte strings.

```php
$s = "\u{200B}hello\u{200B}"; // Zero-width spaces
echo mb_trim($s);              // "hello"
echo mb_ucfirst('über');       // "Über"
```

These fill gaps that have existed since `mbstring` was introduced.

## request_parse_body() for non-POST requests

PHP automatically parses `application/x-www-form-urlencoded` and `multipart/form-data` into `$_POST` and `$_FILES` — but only for POST requests. PATCH and PUT requests with the same content types required manual parsing with `file_get_contents('php://input')` and custom code.

```php
// Inside a PATCH handler
[$_POST, $_FILES] = request_parse_body();
```

The function returns a tuple. The same parsing logic as PHP uses for POST, now accessible for any HTTP method.

## A new DOM API that follows the spec

The existing `DOMDocument` API was based on an older DOM level 3 specification with PHP-specific quirks. 8.4 adds a parallel `Dom\` namespace implementing the WHATWG Living Standard:

```php
$doc = Dom\HTMLDocument::createFromString('<p class="lead">Hello</p>');
$p = $doc->querySelector('p');
echo $p->classList;  // "lead"
echo $p->id;         // ""

$doc2 = Dom\HTMLDocument::createFromFile('page.html');
```

`Dom\HTMLDocument` parses HTML5 correctly — including tag soup. `Dom\XMLDocument` handles strict XML. The new classes are strict about types, return proper node types, and expose modern properties like `classList`, `id`, `className`. The old `DOMDocument` remains, unchanged, for backward compatibility.

## PDO gets driver-specific subclasses

`PDO::connect()` and direct instantiation now return driver-specific subclasses when available:

```php
$pdo = PDO::connect('mysql:host=localhost;dbname=test', 'user', 'pass');
// $pdo is now a Pdo\Mysql instance

$pdo = new Pdo\Pgsql('pgsql:host=localhost;dbname=test', 'user', 'pass');
```

Each driver subclass (`Pdo\Mysql`, `Pdo\Pgsql`, `Pdo\Sqlite`, `Pdo\Firebird`, `Pdo\Odbc`, `Pdo\DbLib`) can expose driver-specific methods without polluting the base `PDO` interface. Doctrine, Laravel, and similar ORMs can now type-hint against driver-specific classes when they need driver-specific behavior.

## OpenSSL gets modern key support

`openssl_pkey_new()` and related functions now support Curve25519 and Curve448 — the modern elliptic curves that have replaced older NIST curves in most security recommendations:

```php
$key = openssl_pkey_new(['curve_name' => 'ed25519', 'private_key_type' => OPENSSL_KEYTYPE_EC]);
$details = openssl_pkey_get_details($key);
```

`x25519` and `x448` for key exchange, `ed25519` and `ed448` for signatures. All four now work with `openssl_sign()` and `openssl_verify()`.

## PCRE: variable-length lookbehind

The bundled PCRE2 library update (10.44) brings variable-length lookbehind assertions — something Perl and Python regex engines had but PHP couldn't do:

```php
// Match "bar" only when preceded by "foo" or "foooo"
preg_match('/(?<=foo+)bar/', 'foooobar', $matches);
```

Previously, lookbehind assertions required a fixed-width pattern. Now they can match patterns of variable length. The `r` modifier (`PCRE2_EXTRA_CASELESS_RESTRICT`) is also new — it prevents mixing ASCII and non-ASCII characters in case-insensitive matches, which closes a class of Unicode confusion attacks.

## DateTime gets microseconds and timestamp factory

```php
$dt = DateTimeImmutable::createFromTimestamp(1700000000.5);
echo $dt->getMicrosecond(); // 500000

$with_micros = $dt->setMicrosecond(123456);
```

`createFromTimestamp()` accepts a float, giving sub-second precision. `getMicrosecond()` and `setMicrosecond()` round out the API for the microsecond component that's been in `DateTime` internally but inaccessible directly.

## fpow() for IEEE 754 compliance

`pow(0, -2)` in PHP has historically returned an implementation-defined value. 8.4 deprecates using `pow()` with a zero base and negative exponent, and introduces `fpow()` which strictly follows IEEE 754 — meaning `fpow(0, -2)` returns `INF`, as the standard defines:

```php
echo fpow(2.0, 3.0);   // 8.0
echo fpow(0.0, -1.0);  // INF
echo fpow(-1.0, INF);  // 1.0
```

Relevant in any code doing mathematical computations where IEEE compliance matters.

## The cost of bcrypt goes up

The default cost for `password_hash()` with `PASSWORD_BCRYPT` increased from `10` to `12`. This affects any code that calls `password_hash($pass, PASSWORD_BCRYPT)` without specifying a cost. The intent is that the default remains roughly "a few hundred milliseconds on modern hardware" as hardware gets faster.

If you store bcrypt hashes and upgrade to 8.4, existing hashes remain valid — `password_verify()` reads the cost from the hash itself. New hashes will use cost 12. `password_needs_rehash()` will return true for old hashes if you pass `['cost' => 12]`, letting you upgrade them on next login.

## Deprecations that matter

Implicitly nullable parameters are gone. If a parameter has a default of `null`, the type must say so explicitly:

```php
// Deprecated in 8.4
function foo(string $s = null) {}

// Correct
function foo(?string $s = null) {}
function foo(string|null $s = null) {}
```

`trigger_error()` with `E_USER_ERROR` is deprecated — it should be replaced with an exception or `exit()`. The `E_USER_ERROR` level was always an awkward hybrid between a recoverable error and a fatal one.

`lcg_value()` is deprecated; use `Random\Randomizer::getFloat()` instead. The LCG generator had poor randomness properties and no seeding control.
