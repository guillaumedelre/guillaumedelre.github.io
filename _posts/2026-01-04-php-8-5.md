PHP 8.5 shipped November 20th. Two features define this release: the pipe operator and the URI extension. They solve different problems, but both share the same motivation — making common operations less awkward to express.

## :arrow_right: The pipe operator

Functional pipelines in PHP have always been verbose. Chaining transformations meant nesting function calls inside out, or breaking them into intermediate variables:

```php
// before — read right to left
$result = array_sum(array_map('strlen', array_filter($strings, 'strlen')));

// or verbose but readable
$filtered   = array_filter($strings, 'strlen');
$lengths    = array_map('strlen', $filtered);
$result     = array_sum($lengths);

// after — read left to right
$result = $strings
    |> array_filter(?, 'strlen')
    |> array_map('strlen', ?)
    |> array_sum(?);
```

The `|>` operator passes the left-hand value into the right-hand expression. The `?` placeholder marks where it's inserted. Pipelines now read in the order operations happen — left to right, top to bottom.

This is particularly useful with first-class callables from PHP 8.1. The two features compose:

```php
$result = $input |> trim(...) |> strtolower(...) |> $this->normalize(...);
```

## :link: The URI extension

Handling URIs in PHP has always required either a third-party library or an uncomfortable combination of `parse_url()` (returns an array, not an object), `http_build_query()`, and manual string concatenation.

The new `Uri` extension provides a proper object-oriented API:

```php
$uri = Uri\Uri::parse('https://example.com/path?query=value#fragment');
$modified = $uri->withPath('/new-path')->withQuery('key=val');
echo $modified; // https://example.com/new-path?key=val#fragment
```

Immutable value objects, RFC-compliant parsing, and the ability to modify individual components without parsing and reconstructing the whole string. Long overdue.

## :no_entry_sign: #[\NoDiscard]

A new attribute that generates a warning when the return value is ignored:

```php
#[\NoDiscard("Use the returned collection, the original is unchanged")]
public function filter(callable $fn): static { ... }
```

Useful for immutable methods where ignoring the return value is almost certainly a bug. The attribute was already common in other languages — PHP 8.5 brings it to the standard library where appropriate.

## :baby: clone with

Cloning an object with modified properties without using property hooks or a custom `with()` method:

```php
$updated = clone($point) with { x: 10, y: 20 };
```

Clean syntax for a pattern that readonly objects in particular needed — you clone to "modify" since you can't mutate.

PHP 8.5 is functional in influence. The pipe operator and URI extension together make code that processes and transforms data significantly more readable. The language continues to evolve in a coherent direction.

## Closures in constant expressions

A constraint that's been in the language since PHP 5: constant expressions — attribute arguments, property defaults, parameter defaults, `const` declarations — couldn't contain closures or first-class callables. 8.5 removes that restriction.

```php
#[Validate(fn($v) => $v > 0)]
public int $count = 0;

const NORMALIZER = strtolower(...);

class Config {
    public function __construct(
        public readonly Closure $transform = trim(...),
    ) {}
}
```

This is the missing piece that makes attributes genuinely expressive for validation and transformation rules. Before 8.5, you had to pass class names or string references to attributes and let the framework look them up — now the callable can live directly in the attribute.

## Attributes on constants

The `#[\Deprecated]` attribute introduced in 8.4 couldn't be applied to `const` declarations. 8.5 adds attribute support for constants generally:

```php
const OLD_LIMIT = 100;

#[\Deprecated('Use RATE_LIMIT instead', since: '3.0')]
const API_TIMEOUT = 30;

const RATE_LIMIT = 60;
```

`ReflectionConstant` — a new reflection class in 8.5 — exposes `getAttributes()` so tools can read them. Combined with closures in constant expressions, attributes on constants become a proper metadata layer for compile-time values.

## #[\Override] extends to properties

PHP 8.3 brought `#[\Override]` for methods. 8.5 extends it to properties:

```php
class Base {
    public string $name = 'default';
}

class Derived extends Base {
    #[\Override]
    public string $name = 'derived';
}
```

If the property doesn't exist in the parent, PHP throws an error. Particularly useful with property hooks introduced in 8.4 — you can now signal that a hooked property is intentionally overriding a parent's.

## Static asymmetric visibility

8.4 introduced asymmetric visibility (`public private(set)`) for instance properties. 8.5 extends it to `static` properties:

```php
class Registry {
    public static private(set) array $items = [];

    public static function register(string $key, mixed $value): void {
        self::$items[$key] = $value;
    }
}

echo Registry::$items['foo']; // readable
Registry::$items['bar'] = 1; // Error: cannot write outside class
```

The pattern is straightforward: expose a static collection for reading, prevent external mutation.

## Constructor promotion for final properties

Property promotion in constructors has existed since PHP 8.0. The `final` modifier on promoted properties was missing — 8.5 adds it:

```php
class ValueObject {
    public function __construct(
        public final readonly string $id,
        public final readonly string $name,
    ) {}
}
```

A class inheriting this can't override `$id` or `$name` with a property of the same name. The combination of `final readonly` on promoted properties makes value objects as sealed as they can be without sealing the class itself.

## Casts in constant expressions

Another gap in constant expressions: type casts weren't allowed. 8.5 allows them:

```php
const PRECISION = (int) 3.7;      // 3
const THRESHOLD = (float) '1.5';  // 1.5
const FLAG = (bool) 1;            // true
```

This sounds minor until you have configuration constants derived from environment variables or other sources that need type coercion at the constant declaration site.

## Fatal errors include backtraces

Before 8.5, a fatal error (out-of-memory, stack overflow, type error in certain contexts) produced a message with no context about where in the code it originated. Finding the culprit meant inserting debug logging and reproducing.

8.5 adds stack backtraces to fatal error messages. They appear in the same format as exception backtraces. A new INI directive, `fatal_error_backtraces`, controls the behavior — it's on by default.

## array_first() and array_last()

PHP has had `reset()` and `end()` for accessing the first and last elements of an array since PHP 3. Both functions mutate the array's internal pointer, which means they're not safe to call on a reference, and they return `false` for empty arrays in a way that's indistinguishable from a stored `false` value.

```php
$values = [10, 20, 30];

$first = array_first($values);  // 10
$last  = array_last($values);   // 30

$first = array_first([]);       // null
```

The new functions return `null` for empty arrays, don't touch the internal pointer, and work on any array expression without a variable. `reset($this->getItems())` was a warning waiting to happen.

## get_error_handler() and get_exception_handler()

PHP has `set_error_handler()` and `set_exception_handler()`. Getting the current handler required either storing it yourself before setting it or calling `set_error_handler(null)` and capturing what it returned — which also cleared the handler.

8.5 adds:

```php
$current = get_error_handler();
$current = get_exception_handler();
```

Useful in middleware chains where you want to wrap the existing handler without losing it, or in testing where you want to verify that a handler was installed.

## IntlListFormatter

Formatting a list with locale-appropriate conjunctions has always required manual string assembly. 8.5 adds `IntlListFormatter`:

```php
$formatter = new IntlListFormatter('en_US', IntlListFormatter::TYPE_AND);
echo $formatter->format(['apples', 'oranges', 'pears']);
// "apples, oranges, and pears"

$formatter = new IntlListFormatter('fr_FR', IntlListFormatter::TYPE_OR);
echo $formatter->format(['rouge', 'bleu', 'vert']);
// "rouge, bleu ou vert"
```

The class wraps ICU's `ListFormatter`. Three types: `TYPE_AND`, `TYPE_OR`, `TYPE_UNITS`. Width constants control whether you get "and" or "&". The Oxford comma handling, the locale-specific conjunction placement — all handled by ICU.

## FILTER_THROW_ON_FAILURE for filter_var()

`filter_var()` returns `false` on validation failure, which produces the classic `false vs null vs 0` ambiguity when filtering untrusted input. A new flag changes this:

```php
try {
    $email = filter_var($input, FILTER_VALIDATE_EMAIL, FILTER_THROW_ON_FAILURE);
} catch (Filter\FilterFailedException $e) {
    // explicitly invalid, not ambiguously false
}
```

The `Filter\FilterFailedException` and `Filter\FilterException` classes are new in 8.5. The flag cannot be combined with `FILTER_NULL_ON_FAILURE` — the behaviors are mutually exclusive.

## Deprecations that clean up years of technical debt

The backtick operator (`` `command` `` as an alias for `shell_exec()`) is deprecated. It's an obscure syntax that surprises readers and is inconsistent with how every other PHP function is called.

Non-canonical cast names — `(boolean)`, `(integer)`, `(double)`, `(binary)` — are deprecated in favor of their short forms: `(bool)`, `(int)`, `(float)`, `(string)`. The long forms have been undocumented for years; 8.5 starts the formal removal process.

Semicolon-terminated `case` statements are deprecated:

```php
// deprecated
switch ($x) {
    case 1;
        break;
}

// correct
switch ($x) {
    case 1:
        break;
}
```

The semicolon form has been syntactically valid since PHP 4 but nobody uses it intentionally. It's a typo that PHP happened to accept.

`__sleep()` and `__wakeup()` are deprecated in favor of `__serialize()` and `__unserialize()`, which return/receive arrays and compose correctly with inheritance. The old methods had problematic semantics around property visibility.

## max_memory_limit caps runaway allocations

A new startup-only INI directive: `max_memory_limit`. It sets an upper bound that `memory_limit` cannot exceed at runtime. If a script calls `ini_set('memory_limit', '10G')`, and `max_memory_limit` is `512M`, PHP warns and caps the value.

Useful in shared hosting environments, or any deployment where you want to ensure that a bug or a malicious payload can't cause a single process to consume the machine's entire RAM by convincing PHP to raise its own limit.

## Opcache is always present

In 8.5, Opcache is always compiled into the PHP binary and always loaded. The previous situation — where Opcache was a loadable extension that might or might not be present depending on build configuration — is gone.

This doesn't mean you can't disable it: `opcache.enable=0` still works. What changes is the guarantee that the Opcache API (`opcache_get_status()`, `opcache_invalidate()`, etc.) is always available, regardless of how PHP was compiled. Code that checks `extension_loaded('opcache')` before calling Opcache functions can drop the check.
