The `compose.yaml` file at the root of the project declared two named volumes:

```yaml
volumes:
    share_storage:
    share_media:
```

These volumes were mounted on several services simultaneously. The `bam` service wrote files to `share_storage`. The `media` service read from `share_media` and wrote thumbnails back to it. The `conversion` service needed both. In Docker Compose, this is invisible — a named volume is just a directory on the host, shared transparently between containers.

In Kubernetes, a volume shared between multiple pods requires a `ReadWriteMany` PersistentVolumeClaim. Not every storage provider supports it. When they do, it tends to be expensive and slow. When they don't, you find out at the worst possible time.

But before any of that became a problem, someone had already built the escape hatch.

## The pattern already in the code

The `media` service has a Flysystem configuration that reads like a catalog:

```yaml
flysystem:
    storages:
        media.storage.local:
            adapter: 'local'
            options:
                directory: "/"

        media.storage.aws:
            adapter: 'aws'
            options:
                client: 'aws_client_service'
                bucket: 'media'
                streamReads: true

        media.storage.azure:
            adapter: 'azure'
            options:
                client: 'azure_client_service'
                container: 'media'

        media.storage:
            adapter: 'lazy'
            options:
                source: '%env(APP__FLYSYSTEM_MEDIA_STORAGE)%'
```

Three concrete adapters — local filesystem, AWS S3, Azure Blob — and one `lazy` adapter on top of them. The lazy adapter doesn't do anything itself; it delegates to whichever adapter the environment variable names.

All application code depends on `media.storage`. It doesn't know, and doesn't need to know, whether its files live on the filesystem, in S3, or in Azure Blob Storage. That decision lives in a single environment variable:

```dotenv
APP__FLYSYSTEM_MEDIA_STORAGE=media.storage.aws
```

This is exactly what <a href="https://12factor.net/backing-services" target="_blank" rel="noopener noreferrer">Factor IV</a> describes: backing services as attached resources, interchangeable through configuration. The same code, pointed at a different endpoint by a different value. No rebuild, no code change.

## What the shared volumes were hiding

The `share_media` volume worked because files written by one container appeared in another container's filesystem at the same path. Two services could share media without knowing about each other. It felt like a filesystem — because it was one.

The problem is that a filesystem isn't a backing service in the twelve-factor sense. It's local state. It's the same class of issue as the cache adapter from [the first article in this series](/2026/05/15/the-cache-that-was-lying-to-us/): invisible on one node, broken on two.

Scale the `media` service to two pods in Kubernetes. Pod A receives an upload, stores it on the local filesystem. Pod B receives the request to serve that file. Pod B has an empty filesystem. The file doesn't exist from Pod B's perspective.

The shared volume pattern masks this problem in Docker Compose because there's only one host, and both containers mount the same directory on that host. The moment you introduce a second node — or a second pod on a different node — the illusion breaks.

With Flysystem and `media.storage.aws`, both pods read and write to the same S3 bucket. The filesystem is gone. <a href="https://12factor.net/processes" target="_blank" rel="noopener noreferrer">Factor VI</a> satisfied — by configuration, not code.

## LiipImagine was already wired

The `media` service generates image thumbnails on demand using LiipImagine. This adds a layer: not only does the original media need to live somewhere accessible, but the generated cache of resized images does too. In a multi-pod setup, a thumbnail generated by Pod A must be findable by Pod B.

The configuration shows that someone had already thought this through:

```yaml
liip_imagine:
    loaders:
        default:
            flysystem:
                filesystem_service: 'media.storage'
        default_cache:
            flysystem:
                filesystem_service: 'media.cache.storage'

    data_loader: 'default'
    cache: 'default_cache'
```

Both the source loader and the cache resolver go through Flysystem. `media.storage` is the lazy adapter for originals. `media.cache.storage` is a separate lazy adapter for generated thumbnails, backed by its own bucket:

```dotenv
APP__FLYSYSTEM_MEDIA_CACHE_STORAGE=media.cache.storage.aws
```

The full pipeline — receive upload, store original, generate thumbnail on first request, cache thumbnail for subsequent requests — is cloud-portable without touching a line of PHP. The transformation is already cloud-native; it just needed to be pointed at the right bucket.

## The dev environment as proof

The local development setup uses <a href="https://min.io/" target="_blank" rel="noopener noreferrer">Minio</a>, an S3-compatible object storage that runs in Docker. The same AWS adapter targets S3 in production and the Minio container in development — only the endpoint changes.

This means the integration between the application and S3 was testable locally before the cloud migration even started. No mocking, no special test adapters, no "it works in prod but not in dev" surprises. The API surface of S3 was already the contract — Minio just emulated it on a laptop.

When cloud credentials replace local Minio credentials in the environment, the code doesn't notice. The bucket responds to the same API. The application doesn't care.

## What remained

The shared volumes were still present in `compose.yaml` for services that hadn't fully cut over. Four services still mount one or both of them:

| Service | Volumes | Flysystem adapters | Status |
|---|---|---|---|
| `media` | `share_storage`, `share_media` | local / aws / azure | cloud-configured, volumes vestigial |
| `bam` | `share_storage` | local / aws / azure | adapters ready, env var to flip |
| `conversion` | `share_storage` | local / aws / azure | adapters ready, env var to flip |
| `sitemap` | `share_media` | local / aws | adapters ready, env var to flip |

For `media`, the Flysystem adapters already point to cloud storage — the volume mounts remain as a fallback for cold storage paths not yet routed through the abstraction. For the other three, the code is ready; the migration is one env var per service.

Unlike the `.dockerignore` in [the secrets article](/2026/05/15/layers-remember-everything/), where the right instinct arrived one step too late, here the abstraction was built before any pressure to use it. The abstraction was built before the migration was planned. That's not common. Usually the decision to go cloud comes first, the abstraction comes under pressure, and the code shows the seams. Here the seams are clean because Flysystem's adapter model is exactly the right shape for this problem: one interface, multiple backends, no business logic that cares which one is active.

The shared volumes in Docker Compose were not a design mistake. They were a reasonable choice for an environment where only one host was running. The mistake would have been assuming that a shared volume is a permanent architecture rather than a local convenience. The lazy adapter is what makes the difference: when the volume is no longer the right answer, the application doesn't need to be told. The environment variable is.
